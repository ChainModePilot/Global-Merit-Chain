# 第十章 繼承與監護

## 10.1 問題背景

人類原型過世後，其累積的 MeriToken 和 iFay 需要妥善處理。核心矛盾：

- 尊重逝者的歷史貢獻
- 防止無關人員取得不相稱的話語權
- 保證「信譽不可世襲」的底線

## 10.2 繼承規則

### 可繼承 vs 不可繼承

| 可繼承 | 不可繼承 |
|--------|----------|
| MeriToken（帶折損） | iFay 的身份 |
| MeritPocket 的監護權 | 以逝者身份行事的權利 |
| coFay 的歸屬權 | iFay 與人類原型的綁定 |

### 折損機制

```
繼承獲得的 MeriToken = 被繼承者 curMerit × 繼承係數
繼承係數 = f(親密度)  ← 親密度越高，折損越少
```

- 親密度極低的人不允許繼承
- 繼承的 MeriToken 也會正常衰減
- 繼承會增加繼承者的 minMerit（但增幅也受折損影響）

### 為什麼需要折損

- MeriToken 代表個人貢獻，繼承者並非創造者
- 無折損繼承會導致「信譽世襲」，違背 GMC 初衷
- 折損比與親密度掛鉤：親密關係本身也是社會貢獻的體現
- MeriToken 本身會衰減，繼承再疊加折損，確保影響快速消退

## 10.3 繼承者身份認證

1. **關係驗證**：透過鏈上社會關係圖譜驗證關係
2. **親密度確認**：確認數值，計算折損比
3. **多方見證**：共同關係人見證確認
4. **冷卻期**：允許異議

### 防繼承欺詐

- 關係必須在被繼承者生前就已上鏈
- 不允許事後補錄
- 親密度基於歷史交互資料，無法臨時偽造

## 10.4 監護

監護 ≠ 繼承身份。監護人可以管理 iFay，但不能以逝者身份行事。

| 監護人可以做 | 監護人不能做 |
|-------------|-------------|
| 管理 iFay 日常運行 | 以逝者身份發表言論 |
| 決定是否進入數位墓園 | 以逝者身份投票 |
| 處理未完成事務 | 以逝者身份取得 Merit |

所有監護行為在鏈上標記操作者為監護人。

## 10.5 數位墓園

- 人類原型過世後 iFay 可被移入
- 仍可能有被動互動，但標註「來自數位墓園」
- 不再主動產生新 MeriToken
- 已有 MeriToken 繼續衰減，最終趨近 minMerit

## 10.6 coFay 的繼承

coFay 作為資產，遵循資產繼承邏輯：
- 歸屬權轉移給繼承者
- MeriToken 不折損（因為貢獻是 coFay 自身產生的）
- 本質區別：繼承的是「資產所有權」，不是「個人信譽」

## 10.7 討論備註

> 繼承機制的設計哲學：
> - 核心矛盾：尊重逝者貢獻 vs 防止信譽世襲
> - 解決方案：允許繼承但強制折損，折損比由客觀親密度決定
> - iFay 不可轉移保證了「人格不可繼承」的底線
> - 數位墓園為 AI 時代的「數位遺產」提供了處理框架
> - coFay 繼承不折損，因為 coFay 是資產而非人格
