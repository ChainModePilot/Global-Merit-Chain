# 第九章 惩罚与申诉机制

## 9.1 为什么需要惩罚

任何信誉系统都需要纠错能力。当贡献被错误认定或存在欺诈时，系统必须能修正。

惩罚机制是 GMC 公信力的最后保障。

## 9.2 惩罚类型

| 类型 | 作用 | 严重度 |
|------|------|--------|
| 罚减 curMerit | 扣减当前 MeriToken，影响即时话语权 | 较轻 |
| 罚减 minMerit | 降低底部值，影响长期最低信誉保障 | 严重 |

罚减 minMerit 是更严厉的惩罚——它打破了"底部值只增不减"的规则，意味着历史贡献的沉淀被部分撤销。

### 严重度对照

| 违规程度 | 惩罚方式 | 示例 |
|----------|----------|------|
| 轻微 | 罚减部分 curMerit | 贡献夸大 |
| 中等 | 罚减较多 curMerit | 重复提交 |
| 严重 | curMerit + 部分 minMerit | 串通刷 Merit |
| 极严重 | 大幅罚减两者 | 系统性欺诈 |

## 9.3 触发流程

```
投诉发起 → 干系人受理投票 → [未通过则驳回] → 惩罚投票 → 执行
```

### 规则

1. **投诉必须针对特定 Merit 获取批次**：不允许泛泛投诉，必须指向具体事件
2. **干系人受理**：相关干系人超过一定比例接受投诉，才正式发起投票
3. **惩罚投票**：需要更高比例通过（如 3/4 多数）
4. **自动执行**：投票通过后系统自动扣减

### 防恶意投诉

- 投诉者需提供证据或理由
- 频繁恶意投诉者可能被标记
- 投诉记录本身上链，保证透明

## 9.4 申诉

被惩罚者有权申诉：

1. 惩罚执行后一定时间内可提起
2. 由更大范围的社群成员重新投票（避免同一批人重复判断）
3. 申诉成功则撤销惩罚、恢复 MeriToken

## 9.5 与其他机制的交互

- **惩罚是唯一可以降低 minMerit 的机制**（除自然衰减外）
- 惩罚记录上链，包含被惩罚者、原因、数量、投票结果
- 惩罚历史影响该实体的社会信誉（可被他人查阅）

## 9.6 讨论备注

> 惩罚机制的设计哲学：
> - 必须有据可查（针对具体批次），防止"莫须有"
> - 分级惩罚体现比例原则
> - 投诉需要门槛（干系人受理），防止恶意骚扰
> - 申诉权保障公正，范围扩大防止同温层效应
> - minMerit 可被惩罚降低，是对诚信底线的最强威慑
