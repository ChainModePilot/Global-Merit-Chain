# 第9章：罰則と申立メカニズム

## 9.1 なぜ罰則が必要なのか

いかなる評判システムも誤りを修正する能力を必要とします。貢献が誤って認定された場合や不正が存在する場合、システムは修正を行えなければなりません。

罰則メカニズムはGMCの信頼性に対する最終的な安全装置です。

## 9.2 罰則の種類

| 種類 | 効果 | 重大度 |
|------|--------|----------|
| curMeritの減額 | 現在のMeriTokenを減少させ、即時の発言権に影響 | 軽度 |
| minMeritの減額 | フロア値を引き下げ、長期的な最低評判保証に影響 | 重度 |

minMeritの減額はより重い罰則です — 「フロア値は増加のみで減少しない」というルールを破ることを意味し、歴史的貢献の蓄積が部分的に取り消されます。

### 重大度の参考

| 違反レベル | 罰則方法 | 例 |
|-----------------|----------------|---------|
| 軽微 | curMeritの一部を減額 | 貢献の誇張 |
| 中程度 | curMeritの大幅な減額 | 重複提出 |
| 重大 | curMerit + minMeritの一部 | Merit獲得のための共謀 |
| 極めて重大 | 両方の大幅な減額 | 組織的不正 |

## 9.3 トリガープロセス

```
申立提出 → ステークホルダー受理投票 → [不通過の場合は却下] → 罰則投票 → 執行
```

### ルール

1. **申立は特定のMerit取得バッチを対象としなければなりません**：曖昧な申立は許可されず、特定のイベントを指し示す必要があります
2. **ステークホルダー受理**：正式な投票が開始される前に、一定割合の関連ステークホルダーが申立を受理する必要があります
3. **罰則投票**：より高い可決閾値が必要です（例：3/4の多数決）
4. **自動執行**：投票が通過すると、システムが自動的に減額を適用します

### 悪意ある申立の防止

- 申立者は証拠または正当な理由を提供しなければなりません
- 頻繁に悪意ある申立を行う者はフラグ付けされる場合があります
- 申立記録自体がオンチェーンに保存され、透明性を確保します

## 9.4 申立（アピール）

罰則を受けた当事者には申立の権利があります：

1. 罰則執行後の一定期間内に申立を提出できます
2. より広範なコミュニティメンバーのグループが再投票します（同じグループが繰り返し判断することを避けるため）
3. 申立が成功した場合、罰則は取り消されMeriTokenが復元されます

## 9.5 他のメカニズムとの相互作用

- **罰則はminMeritを減少させることができる唯一のメカニズムです**（自然減衰を除く）
- 罰則記録はオンチェーンに保存され、罰則対象エンティティ、理由、金額、投票結果を含みます
- 罰則履歴はエンティティの社会的評判に影響します（他者が閲覧可能）

## 9.6 議論メモ

> 罰則メカニズムの設計哲学：
> - 証拠に基づく必要があります（特定のバッチを対象）、「根拠のない告発」を防止します
> - 段階的な罰則は比例原則を反映します
> - 申立には閾値が必要です（ステークホルダー受理）、悪意あるハラスメントを防止します
> - 申立の権利は公平性を保護します。範囲の拡大はエコーチェンバー効果を防ぎます
> - minMeritが罰則によって減少可能であることが、誠実性違反に対する最も強力な抑止力として機能します
